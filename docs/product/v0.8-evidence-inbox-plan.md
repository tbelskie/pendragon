# v0.8 Evidence Inbox Plan

The next integration move should stay lean.

Do not build OAuth, credential storage, API polling, or bring-your-own-credentials ingestion yet. Those are real future paths, but they create security and support obligations before Pendragon has proven that Forge output is worth the work.

v0.8 creates the smaller primitive:

> Paste the source, label what it proves, and let Forge use it.

## Product Bet

Founders already keep launch truth in scattered places:

- GitHub repos
- Google Docs
- Notion pages
- pricing docs
- analytics dashboards
- support forms
- demo videos
- app store listings
- waitlists

Pendragon does not need to connect to all of them yet. It needs to capture the evidence those sources represent.

## v0.8 Scope

Ship a local-only Evidence tab with:

- title
- link
- type
- attachment target
- proof note

Forge then treats safe evidence links attached to Forge as launch proof.

## Evidence Types

Initial types:

- repo
- doc
- pricing
- analytics
- design
- support
- demo
- waitlist
- store
- other

## Attachment Targets

Keep the first attachment model tiny:

- Brief
- Docs
- Gate
- Forge

This lets founders say whether a source supports positioning, docs, readiness, or the generated launch surface.

## Non-Goals

Do not add:

- OAuth
- API keys
- token storage
- webhooks
- sync jobs
- generic data pipelines
- scraping
- file upload
- automatic summarization

Those belong after the manual evidence loop proves useful.

## Success Criteria

- A founder can add an evidence link without leaving the product room.
- The evidence link is saved locally.
- The Evidence tab shows total links, Forge proof links, and trust signals.
- Forge uses evidence links attached to Forge as proof cards.
- HTML export includes those evidence cards.
- Writing Quest starts with a small evidence set.

## Future Connector Path

Evidence creates the contract future connectors must satisfy:

> What does this source prove, and where should Pendragon use it?

Later connectors should feed this same model instead of inventing their own surfaces.

Possible future sequence:

1. Link recognition for common URLs.
2. Public metadata fetch for GitHub repos and public pages.
3. Local runner for sensitive credentials.
4. Bring-your-own-source for advanced users.
5. First OAuth connector only when manual evidence capture hurts.
