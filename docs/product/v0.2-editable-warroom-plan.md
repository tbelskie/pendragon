# Pendragon v0.2 Editable Warroom Plan

v0.2 turns the static Pendragon scaffold into a local-first product surface Tom can use while launching Writing Quest.

The goal is not to build a full project management app. The goal is to make one product room editable, trustworthy, and polished enough that it becomes useful in weekly launch work.

## Product Job

Pendragon v0.2 should answer, and let the founder update:

> What matters this week, what actions move it forward, what decision is blocking us, and what risk could sink the launch?

## User

The first user is Tom dogfooding Pendragon against the four-product portfolio.

The first product room is Writing Quest.

## Product Principle

Keep the warroom calm.

Editing should feel like updating a premium command surface, not filling out an admin form.

## Required Scope

### Editable Warroom Fields

The active product room must support editing:

- current focus
- next three actions
- decision needed
- top risk
- readiness counts for Build, Docs, Launch, Trust, and Revenue

### Local Persistence

- Save edited product data to browser local storage.
- Preserve the active selected product.
- Keep the current `pendragon.workspace.v1` storage shape unless the data contract changes.
- Add a visible saved state so the user trusts the local-only prototype.

### Reset

- Keep the existing reset-to-demo behavior.
- Make reset discoverable from the interface, not only a keyboard shortcut.
- Reset must be clearly labeled as destructive to local edits.

### Export

- Add a lightweight JSON export path.
- Export should download or copy the current local workspace.
- Import is not required for v0.2 unless it falls out naturally.

### Premium Front-End Feel

- The default warroom must remain the primary experience.
- Editing should be a mode or drawer, not a page of raw inputs.
- Buttons should look like deliberate product controls.
- Text fields must inherit the Lava Glass visual system.
- Mobile must remain usable.

## Explicit Non-Goals

- Auth
- Backend
- Billing
- AI generation
- GitHub integration
- Forge generation
- Setup flow
- Multi-user collaboration
- Drag and drop
- Rich text
- Full task management
- Full decision ledger
- Full docs asset tracker

## UX Contract

### Default Mode

Default mode is for reading the room.

It should show:

- selected product
- stage mode
- launch readiness score
- this week's launch focus
- next three actions
- readiness panel
- decision needed
- top risk
- Forge placeholder

Primary controls:

- Edit room
- Export JSON
- Reset demo

### Edit Mode

Edit mode is for sharpening the room.

It should show a contained editor for the active product with:

- text input for current focus
- three text inputs for next actions
- text input or textarea for decision needed
- text input or textarea for top risk
- compact numeric controls for readiness counts
- Save changes
- Cancel

Save should be explicit. This avoids the "did it save?" anxiety that local-only apps often create.

### Saved State

After saving, show a small confirmation:

- "Saved locally"
- timestamp or simple state is acceptable

The copy must make clear that data is local-only for now.

### Error State

If local storage save fails:

- keep the edited form values on screen
- show a clear inline error
- do not silently discard work

## Data Contract

Use the existing workspace shape:

```js
{
  activeProductId: "writing-quest",
  products: [
    {
      id: "writing-quest",
      name: "Writing Quest",
      stage: "beta",
      status: "Paid beta launch",
      targetDate: "October 2026",
      oneLiner: "...",
      user: "...",
      currentFocus: "...",
      decisionNeeded: "...",
      topRisk: "...",
      nextActions: ["...", "...", "..."],
      readiness: [
        { name: "Build", done: 6, total: 9 }
      ]
    }
  ]
}
```

v0.2 should not add a backend-shaped abstraction layer prematurely.

## Implementation Plan

1. Add this plan and design mockups.
2. Add small workspace update helpers.
3. Add an edit mode state to the app shell.
4. Render an editor for the active product.
5. Save edits into local storage.
6. Add visible save, cancel, reset, and export controls.
7. Verify desktop layout.
8. Verify mobile layout.
9. Merge behind the existing `/app/` URL.

## Acceptance Criteria

- The app still opens at `/app/`.
- The real Pendragon logo remains in the shell.
- Switching products still works.
- Editing Writing Quest and saving persists after reload.
- Cancel does not mutate saved data.
- Reset restores demo seed data.
- Export returns a valid JSON representation of the workspace.
- Mobile has no page-level horizontal overflow.
- Browser console has no errors during basic flows.
