# Operating Model

Pendragon is a working system, not a thought exercise.

## Weekly Rhythm

Each week:

1. Review the active product room.
2. Update the launch room.
3. Record decisions.
4. Choose one build priority.
5. Choose one docs or UX writing priority.
6. Choose one GTM experiment.
7. End with a short retrospective.

## Product Room Contents

Each product should maintain:

- product brief
- launch room
- docs plan
- pricing hypothesis
- GTM experiment log
- decision ledger
- current risks
- next actions
- retrospective

## Decision Discipline

Decisions should record:

- date
- product
- decision
- context
- options considered
- why this path
- revisit trigger

The goal is to stop re-litigating the same questions without new evidence.

## Launch Readiness

Before launch, every product needs:

- clear buyer/user
- clear offer
- pricing or free strategy
- onboarding path
- support/contact path
- docs/help content
- privacy/trust basics
- launch assets
- analytics or feedback loop
- known risks
- post-launch review date

## Kill, Pause, Continue

Each product should have explicit criteria for:

- continue: signal is strong enough to invest another cycle
- pause: signal is weak or timing is wrong
- kill: effort no longer matches expected upside

Pendragon exists partly to make these calls less emotional.
